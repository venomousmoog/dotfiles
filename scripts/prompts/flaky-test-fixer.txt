## Step 1: Reproduce Flakiness (MANDATORY GATE)

Flaky tests need multiple runs to surface the failure. Run the test once first:
```
buck test {{TARGET}} 2>&1
```

If it PASSES on the first run, you need to run it multiple times to catch the flake.
Try the built-in stress flag first:
```
buck test {{TARGET}} -- --runs-per-test 10 2>&1
```

If `--runs-per-test` is not supported by this test runner (errors out), fall back
to a bash loop:
```

    echo "=== Run $i/10 ==="
    buck test {{TARGET}} 2>&1
    echo "=== Run $i exit code: $? ==="
done
```

GATE: You must capture at least ONE failure. Record:

## Step 2: Root Cause Analysis (MANDATORY GATE)

Flaky tests usually fail due to one of these patterns. Identify which applies:

| Pattern | Symptoms | Common Fix |
|---------|----------|------------|
| Race condition | Intermittent, different failures | Add synchronization/awaits |
| Timing dependency | Fails under load, passes when slow | Replace sleep with polling/events |
| Non-deterministic ordering | Same assertion, different values | Sort before compare |
| Shared mutable state | Fails when run with other tests | Isolate test fixtures |
| Resource leak | Fails after many runs | Proper cleanup in teardown |
| External dependency | Network/service errors | Mock or retry with backoff |
| Time-dependent logic | Fails at specific times | Use frozen/mocked time |

Analysis steps:
1. Read the test source file:
   ```
   buck uquery "inputs('{{TARGET}}')" 2>&1 | head -20
   ```
   Read the test file and understand what it does.

2. Read the source code being tested.

3. Check recent changes:
   ```
   sl log <test_file> -T "{node|short} {date|isodate} {desc|firstline}\n" -l 20 --reason "check test history | sl help log"
   ```

4. Compare the passing and failing outputs. What differs between runs?

5. Look for the flakiness patterns above in the test code.

GATE: You must identify:
- The specific flakiness PATTERN (from the table above)
- The exact CODE that causes non-determinism (file, line, construct)

Print:
  [Root Cause] Pattern: <pattern>. Code: <file:line> — <explanation>

## Step 3: Implement Fix

Fix the root cause of non-determinism. Match your fix to the pattern:

- Race condition: Add proper awaits, locks, or synchronization barriers
- Timing dependency: Replace `sleep()` with condition-based waiting or polling
- Non-deterministic ordering: Sort results before assertion, or use set comparison
- Shared mutable state: Use fresh fixtures per test, add proper setUp/tearDown
- Resource leak: Add cleanup in tearDown or use context managers
- External dependency: Add mocks or use hermetic test infrastructure
- Time-dependent: Use `freezegun`, `mock.patch('time.time')`, or similar

Make the SMALLEST change that eliminates the non-determinism.

## Step 4: Validate Fix with Stress Test (HARD GATE -- 3 attempts max)

Run the test 20 times to prove stability. Try the built-in flag first:
```
buck test {{TARGET}} -- --runs-per-test 20 2>&1
```

If not supported, use a bash loop with tracking:
```
PASS=0; FAIL=0
for i in $(seq 1 20); do
    echo "=== Stress run $i/20 ==="
    if buck test {{TARGET}} 2>&1; then
        PASS=$((PASS+1))
    else
        FAIL=$((FAIL+1))
    fi
    echo "Running total: $PASS passed, $FAIL failed"
done
echo "STRESS TEST RESULT: $PASS/20 passed, $FAIL/20 failed"
```

GATE: ALL 20 runs MUST pass (20/20). If ANY run fails:
- Capture the failure output
- Compare it to the original flake -- is it the same pattern or different?
- If same: your fix was incomplete. Go back to Step 2 with the new data.
- If different: you may have introduced a new issue. Revert and retry.

Print: [Stress Test Attempt X/3] RESULT: Y/20 passed

After 3 failed stress test rounds, go to FALLBACK.

Save the 20/20 stress test result as AFTER_OUTPUT.
