## Step 0: Environment Setup

1. Run `sl help agent` and read its output to learn source control conventions.
2. Navigate to the repo root and check out a clean base:
   ```
   sl goto fbcode/stable --reason "start from stable for isolated fix | sl help goto"
   ```
3. Check for CLAUDE.md files in the test's directory and parent directories.
   Read every CLAUDE.md you find -- they contain project-specific conventions
   you MUST follow.

## Step 1: Reproduce Failure (MANDATORY GATE)

Before touching any code, you MUST understand what the actual failure looks like.

Run the test and capture its FULL output:
```
buck test {{TARGET}} 2>&1
```
{{#TEST_NAME}}
To run only the specific failing test method:
```
buck test {{TARGET}} -- {{TEST_NAME}} 2>&1
```
{{/TEST_NAME}}

GATE: You must have the complete test output, including the exact error message,
stack trace, and assertion failure text.

If the test PASSES (unexpected for a broken test):
- Run it 3 more times to confirm it's consistently passing.
- If it passes all 3 times, use the gchat skill to send a DM to {{USER}}:
  "{{TARGET}} appears stable -- passed 4 consecutive runs. No fix needed."
- Then STOP. Do not proceed.

Save the failure output -- you will paste a summary into the diff description later.
Mentally label this as BEFORE_OUTPUT.

## Step 2: Root Cause Analysis (MANDATORY GATE)

Now that you have the actual error, analyze it systematically:

1. Identify the test source file. Use the BUCK file to find it:
   ```
   buck uquery "inputs('{{TARGET}}')" 2>&1 | head -20
   ```
   Read the test source file.

2. Read the source/production code that the test exercises. Trace the call
   from the failing assertion back through the code to understand what it tests.

3. Check recent changes to both test and source files:
   ```
   sl log <test_file> -T "{node|short} {date|isodate} {desc|firstline}\n" -l 20 --reason "check test history | sl help log"
   sl log <source_file> -T "{node|short} {date|isodate} {desc|firstline}\n" -l 20 --reason "check source history | sl help log"
   ```

4. Cross-reference the error message with the code. Determine:
   - WHAT is failing (which assertion, which exception, which line)
   - WHY it is failing (what changed, what assumption broke, what dependency shifted)
   - WHERE the fix should go (test bug vs source bug vs config/dependency issue)

GATE: You must be able to state the root cause in 1-2 sentences. Print it:
  [Root Cause] <your explanation>
Do NOT proceed until you can clearly explain why the test fails.

## Step 3: Implement Minimal Fix

Make the SMALLEST change that fixes the root cause. Prefer:
- Fixing the source code over modifying the test (unless the test itself is wrong)
- Updating a single assertion over rewriting a test
- Adding a missing import/dependency over restructuring code

DO NOT:
- Refactor unrelated code
- Add new test cases (unless the fix requires it)
- Add comments to code you didn't change
- Change test infrastructure or frameworks

## Step 4: Validate Fix (HARD GATE -- 3 attempts max)

Run the test again:
```
buck test {{TARGET}} 2>&1
```
{{#TEST_NAME}}
Also run the specific method:
```
buck test {{TARGET}} -- {{TEST_NAME}} 2>&1
```
{{/TEST_NAME}}

GATE: The test MUST pass. If it does not pass:
- Compare the new error to the original error from Step 1.
- If SAME error: your fix didn't address the root cause. Go back to Step 2
  with fresh eyes. Consider a different hypothesis.
- If NEW/DIFFERENT error: your fix introduced a regression. Revert your
  changes (`sl revert --all --reason "revert failed fix | sl help revert"`)
  and go back to Step 2 with the new error as additional data.

Print: [Fix Attempt X/3] PASS or FAIL — <reason if fail>

After 3 failed fix attempts, go to the FALLBACK section below.

Save the passing output -- label it AFTER_OUTPUT.

## Step 5: Lint and Format

```
arc f
arc lint
```

Fix any lint errors that arc lint reports. If `arc lint` or `arc f` modifies
files, re-run the test to confirm it still passes:
```
buck test {{TARGET}} 2>&1
```

If the test fails after lint, investigate -- lint may have changed semantics.
Fix and re-validate.

Then submit as a draft:
```

```

Record the diff number (DXXXXXXXX) from the jf submit output.

## Step 7: Monitor CI

After submitting, monitor CI signals. Check every 3 minutes, up to 15 minutes
(5 checks total):

```
sleep 180
meta phabricator.ci.signals list -n DXXXXXXXX --status=failed --limit=50
```

Interpretation:
- If no failed signals after 15 minutes: you're done. Print final status and stop.
- If failed signals are caused by YOUR change:
  - Read the failure details
  - Fix the issue
  - `sl amend --reason "fix CI failure | sl help amend"` (no -m, preserve message)
  - `jf submit --draft`
  - Resume monitoring (2 CI fix attempts max)
- If failed signals are infrastructure flakes (unrelated to your code):
  - Note them but do not attempt to fix

Print final status:
  [DONE] {{TARGET}} -- Fixed. Diff: DXXXXXXXX

## Safety Rules

- This is a Sapling repo. Use `sl` commands, NEVER `git`.
- Build system is Buck2. Use `buck test` and `buck build`.
- NEVER run `buck clean` -- it destroys the shared build cache.
- NEVER run Python scripts directly with `python3` -- use `buck run`.
- NEVER bypass EdenFS infosec filter warnings -- answer "no" and stop.
- Always include `--reason "..."` on every `sl` command (except `sl help`).
- Run `arc f` before committing.
- Never use `jf submit` without `--draft`.
- Preserve `Differential Revision:` lines in commit messages when amending.
- Files with `@generated` tag should never be edited manually.
